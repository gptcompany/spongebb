### Phase 23: Workspace Backend Integration

**Goal**: Expose existing FastAPI as OpenBB Workspace custom backend via `openbb-api --app`
**Depends on**: Phase 9 (FastAPI API), Phase 10 (Dashboard), Phase 22 (Container runtime)
**Plans**: 4 plans

Plans:
- [x] 23-01: Foundation — `openbb-platform-api` dependency, version pin, CORS, `workspace_app.py`
- [x] 23-02: Workspace metric + chart endpoints (`/workspace/*` routes)
- [x] 23-03: Table endpoint annotations (`openapi_extra` widget_config) + Docker service
- [x] 23-04: Integration tests + smoke test end-to-end

**Key decisions:**
- Flat Pydantic models need no dataKey; nested models use dataKey to point at array/dict field
- Calendar events uses columnsDefs for structured table display (date pinned, renderFn for type/impact)
- Docker workspace service on port 6900 (profiles: [workspace], host.docker.internal defaults)

### Phase 24: Widget Polish & Optimization

**Goal**: Optimize widget metadata, refresh intervals, column formatting, and auth for production UX
**Depends on**: Phase 23
**Plans**: 3 plans

Plans:
- [x] 24-01: Refresh intervals + stale time per widget type
- [x] 24-02: Column definitions + formatters + renderers
- [x] 24-03: Dynamic date defaults + optional API key auth

**Key decisions:**
- refetchInterval/staleTime tuned per data source frequency (FRED weekly: 4h, NY Fed daily: 15m, PBoC monthly: disabled)
- columnsDefs (with 's') is the correct OpenBB Workspace key
- formatterFn: int, none, percent, normalized, normalizedPercent, dateToYear
- renderFn: greenRed, titleCase, columnColor, hoverCard, cellOnClick, showCellChange

### Phase 25: Native Provider Extension

**Goal**: Create `obb.liquidity.*` Python SDK commands via thin Fetcher adapters wrapping existing calculators
**Depends on**: Phase 23, Phase 24
**Plans**: 3 plans

Plans:
- [x] 25-01: Extension scaffold (`src/liquidity/openbb_ext/`, Provider registration, entry point)
- [x] 25-02: Fetcher adapters (NetLiquidity, GlobalLiquidity, StealthQE)
- [x] 25-03: Tests & verification (17 unit tests: 6 discovery + 11 fetcher TET)

**Key decisions:**
- Lazy imports in `aextract_data()` to avoid circular dependency: provider → calculators → collectors → `from openbb import obb` → provider
- `date as dateType` alias pattern avoids Pydantic field name clash with type annotation
- 3 Fetcher classes follow TET pipeline pattern (Transform-Extract-Transform)
