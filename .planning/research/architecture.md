## 1. OpenBB Platform Architecture Overview

The OpenBB Platform uses a three-layer architecture. Understanding it is the prerequisite for all integration decisions.

```
┌─────────────────────────────────────────────────────────────┐
│                    Client Surfaces                          │
│         Python SDK  │  REST API  │  Workspace UI           │
└───────────────────────────────────────────────────────────┬─┘
                                                            │
┌───────────────────────────────────────────────────────────┴─┐
│                      openbb-core                            │
│   FastAPI base │ OBBject serialization │ Registry/builder   │
└───────────────────────────────────────────────────────────┬─┘
                                                            │
┌───────────────────────────────────────────────────────────┴─┐
│                  Router Extensions                          │
│    openbb-equity  │  openbb-economy  │  openbb-liquidity   │
│          (define endpoints, not data sources)               │
└───────────────────────────────────────────────────────────┬─┘
                                                            │
┌───────────────────────────────────────────────────────────┴─┐
│                  Provider Extensions                        │
│    openbb-fred  │  openbb-yfinance  │  (your providers)    │
│         (outermost layer — actually fetches data)           │
└─────────────────────────────────────────────────────────────┘
```

**Source:** [OpenBB Architecture Blog](https://openbb.co/blog/exploring-the-architecture-behind-the-openbb-platform)

The TET (Transform → Extract → Transform) pipeline is the core pattern every provider Fetcher must implement:

1. **Transform (query):** Validate + transform user input into `QueryParams`
2. **Extract:** Call the external API / data source, return raw data
3. **Transform (data):** Validate raw data into typed `Data` model, return `OBBject`

## 2. Package Structure Decision: Monorepo Subfolder

### Recommendation: Monorepo subfolder — NOT a separate package (for now)

**Rationale:**

The project already has 174 files and 52K LOC. A separate PyPI package (`openbb-liquidity`) requires:
- Separate versioning/release cycle
- Separate CI pipeline
- Entry point registration + `openbb-build` step on every install
- Added complexity when the business logic (`calculators/`) is in another repo

A monorepo subfolder keeps business logic co-located, avoids the circular dependency trap, and can be extracted to a standalone package in v6.0 once the API surface is stable.

### Recommended structure (within existing monorepo):

```
spongebb/                          ← existing repo root
├── src/
│   └── liquidity/                         ← existing main package (unchanged)
│       ├── calculators/                   ← UNCHANGED — business logic stays here
│       ├── collectors/                    ← UNCHANGED
│       ├── api/                           ← UNCHANGED — existing FastAPI routers
│       │
│       └── openbb_ext/                    ← NEW: OpenBB integration layer
│           ├── __init__.py
│           ├── provider.py                ← Provider instance registration
│           ├── workspace_app.py           ← FastAPI app for Workspace backend
│           ├── widgets.json               ← Auto-generated by openbb-api (don't handwrite)
│           └── models/                    ← Fetcher + Data models
│               ├── __init__.py
│               ├── net_liquidity.py       ← NetLiquidityFetcher
│               ├── global_liquidity.py    ← GlobalLiquidityFetcher
│               └── stealth_qe.py         ← StealthQEFetcher
│
├── pyproject.toml                         ← Add openbb_provider_extension entry point
└── tests/
    └── openbb_ext/                        ← New test directory for extension
        ├── cassettes/                     ← vcrpy HTTP cassettes (committed)
        ├── test_net_liquidity_fetcher.py
        ├── test_global_liquidity_fetcher.py
        └── test_stealth_qe_fetcher.py
```

**When to extract to standalone package:** When the extension has a stable API surface, when other teams want to install `openbb-liquidity` independently, or for public PyPI publication.

## 3. Workspace Backend vs Provider Extension — When to Use Each

These two approaches serve different purposes and CAN coexist.

| Dimension | Provider Extension | Workspace Backend |
|-----------|-------------------|-------------------|
| Purpose | Add data source to OpenBB Platform pipeline | Expose custom widgets to OpenBB Workspace UI |
| Python accessible | Yes, via `obb.liquidity.net()` | No — UI only |
| Workspace widgets | Auto-exposed via Platform backend | Direct via widgets.json |
| AI agent support | Indirect | Direct |
| Tech requirement | Python + openbb-core conventions | Any web API (FastAPI works) |
| Config | pyproject.toml entry points + openbb-build | widgets.json + API endpoint |
| Standardization | Required (TET pipeline, Data model) | None — you control the format |

**For this project, build BOTH:**

1. **Provider Extension** — Makes `obb.liquidity.net()`, `obb.liquidity.global_index()`, `obb.liquidity.stealth_qe()` available in the Python SDK and REST API. This is the "proper" integration into the OpenBB ecosystem.

2. **Workspace Backend** — Exposes the existing FastAPI (`src/liquidity/api/`) as an OpenBB Workspace custom backend via `openbb-platform-api`. This is low-effort (the FastAPI already exists) and enables the Workspace UI immediately. It runs alongside the provider extension — they are different entry points.

**Source:** [Data Integration | OpenBB Workspace Docs](https://docs.openbb.co/workspace/developers/data-integration), [Build Provider Extensions](https://docs.openbb.co/python/developer/extension_types/provider)

## 5. Adapter Pattern: Wrapping Calculators into Fetchers

The key design constraint: **zero duplication of business logic**.

The calculators already implement the Extract step (via collectors) and the second Transform step (unit conversion, delta calculation, sentiment). The Fetcher becomes a thin adapter.

### Pattern: Thin Fetcher Adapter

```python
# src/liquidity/openbb_ext/models/net_liquidity.py

from datetime import date
from typing import Any, Optional

from openbb_core.provider.abstract.data import Data
from openbb_core.provider.abstract.fetcher import Fetcher
from openbb_core.provider.abstract.query_params import QueryParams
from pydantic import Field

# ─── QueryParams ──────────────────────────────────────────────
class NetLiquidityQueryParams(QueryParams):
    """Query parameters for Hayes Net Liquidity Index."""

    start_date: Optional[date] = Field(
        default=None,
        description="Start date for time series (YYYY-MM-DD).",
    )
    end_date: Optional[date] = Field(
        default=None,
        description="End date for time series (YYYY-MM-DD).",
    )
    use_daily_series: bool = Field(
        default=False,
        description="Use daily FRED series (WTREGEN, RRPONTSYD) instead of weekly.",
    )


# ─── Data Model ───────────────────────────────────────────────
class NetLiquidityData(Data):
    """Hayes Net Liquidity Index data point.

    All monetary values in billions USD.
    Net Liquidity = WALCL - TGA - RRP
    """

    date: date = Field(description="Observation date.")
    net_liquidity: float = Field(description="Net Liquidity in billions USD.")
    walcl: float = Field(description="Fed Total Assets in billions USD.")
    tga: float = Field(description="Treasury General Account in billions USD.")
    rrp: float = Field(description="Reverse Repo in billions USD.")
    weekly_delta: Optional[float] = Field(
        default=None,
        description="7-day change in billions USD (latest observation only).",
    )
    sentiment: Optional[str] = Field(
        default=None,
        description="Sentiment: BULLISH, NEUTRAL, or BEARISH (latest only).",
    )


# ─── Fetcher ──────────────────────────────────────────────────
class NetLiquidityFetcher(
    Fetcher[
        NetLiquidityQueryParams,
        list[NetLiquidityData],
    ]
):
    """Fetcher for Hayes Net Liquidity Index.

    Delegates computation to NetLiquidityCalculator — does NOT duplicate logic.
    """

    @staticmethod
    def transform_query(params: dict[str, Any]) -> NetLiquidityQueryParams:
        """Validate and transform query parameters."""
        return NetLiquidityQueryParams(**params)

    @staticmethod
    async def aextract_data(
        query: NetLiquidityQueryParams,
        credentials: Optional[dict[str, str]],
        **kwargs: Any,
    ) -> list[dict[str, Any]]:
        """Delegate to NetLiquidityCalculator — single source of truth."""
        from datetime import datetime, UTC, timedelta
        # Import lazily to avoid circular imports and speed up load time
        from liquidity.calculators.net_liquidity import NetLiquidityCalculator
        from liquidity.config import get_settings

        settings = get_settings()
        # Inject FRED API key from OpenBB credentials if provided
        if credentials and credentials.get("fred_api_key"):
            settings = settings.model_copy(
                update={"fred_api_key": credentials["fred_api_key"]}
            )

        calc = NetLiquidityCalculator(
            settings=settings,
            use_daily_series=query.use_daily_series,
        )

        start = (
            datetime.combine(query.start_date, datetime.min.time(), tzinfo=UTC)
            if query.start_date
            else datetime.now(UTC) - timedelta(days=365)
        )
        end = (
            datetime.combine(query.end_date, datetime.min.time(), tzinfo=UTC)
            if query.end_date
            else datetime.now(UTC)
        )

        df = await calc.calculate(start_date=start, end_date=end)

        if df.empty:
            return []

        # Get current for deltas and sentiment (latest row enrichment)
        current = await calc.get_current()
        latest_ts = df["timestamp"].max()

        records = []
        for _, row in df.iterrows():
            is_latest = row["timestamp"] == latest_ts
            records.append({
                "date": row["timestamp"].date() if hasattr(row["timestamp"], "date") else row["timestamp"],
                "net_liquidity": float(row["net_liquidity"]),
                "walcl": float(row["walcl"]),
                "tga": float(row["tga"]),
                "rrp": float(row["rrp"]),
                "weekly_delta": current.weekly_delta if is_latest else None,
                "sentiment": current.sentiment.value if is_latest else None,
            })

        return records

    @staticmethod
    def transform_data(
        query: NetLiquidityQueryParams,
        data: list[dict[str, Any]],
        **kwargs: Any,
    ) -> list[NetLiquidityData]:
        """Validate extracted data against the Data model."""
        return [NetLiquidityData.model_validate(record) for record in data]
```

### Key design rule: lazy imports inside `aextract_data`

The official OpenBB docs explicitly state: "Always lazy-load imports inside functions to avoid circular imports and to speed up initial load time." This is mandatory for our case since `liquidity` package already depends on `openbb`.

## 6. Data Model Mapping: Existing Pydantic → OpenBB Data

OpenBB's `Data` base class is itself a Pydantic model. The mapping is straightforward.

| Existing Model | Fields | OpenBB Data Model | Notes |
|----------------|--------|-------------------|-------|
| `NetLiquidityResult` (dataclass) | timestamp, net_liquidity, walcl, tga, rrp, weekly_delta, monthly_delta, delta_60d, delta_90d, sentiment | `NetLiquidityData(Data)` | Flatten datetime → date for OpenBB convention; expose deltas as optional fields |
| `GlobalLiquidityResult` (dataclass) | timestamp, total_usd, fed_usd, ecb_usd, boj_usd, pboc_usd, boe_usd, snb_usd, boc_usd, weekly_delta, delta_30d, delta_60d, delta_90d, coverage_pct | `GlobalLiquidityData(Data)` | One row = one date; tier2 fields are Optional |
| `NetLiquidityResponse` (Pydantic) | value, walcl, tga, rrp, weekly_delta, sentiment, as_of_date, metadata | `NetLiquidityData(Data)` | Strip metadata wrapper; OpenBB adds its own OBBject wrapper |
| `RegimeResponse` (Pydantic) | regime, intensity, confidence, components, as_of_date | `RegimeData(Data)` | Direct field mapping; `components` string remains |
| `StealthQEResponse` (Pydantic) | score, status, rrp_level, rrp_velocity, tga_level, tga_spending, fed_total, fed_change | `StealthQEData(Data)` | Snapshot only (no time series) |

**Mapping rule:** OpenBB `Data` extends `pydantic.BaseModel`. Your existing models map cleanly because they already use Pydantic or dataclasses. The main difference is:
- `Data` subclasses use `date` (not `datetime`) by convention for the date field
- Metadata wrappers (`APIMetadata`) are dropped — OpenBB provides its own `OBBject` wrapper
- `Optional` fields are fine (OpenBB handles them correctly)

## 9. Dependency Management and Circular Dependency Prevention

### The trap

The project currently has `openbb>=4.0.0` as a dependency in `pyproject.toml`. If the provider extension is registered via entry point and installed as part of the same package, there is a potential circular dependency:

```
openbb (meta-package) → installs provider extensions
provider extension → depends on openbb → installs provider extensions → ...
```

### The fix (two-part)

**Part 1: The extension layer depends on `openbb-core`, not `openbb`**

`openbb-core` is the base package that does NOT depend on provider extensions. The meta-package `openbb` bundles `openbb-core` + specific providers. Since the extension uses `openbb-core` types (`Fetcher`, `Data`, `QueryParams`, `Provider`), declaring `openbb-core` as the dependency is correct and avoids the cycle.

The existing `liquidity-monitor` package that depends on `openbb>=4.0.0` is fine — it's not a provider extension itself. Only `liquidity.openbb_ext` acts as a provider extension, and it only needs `openbb-core` types.

```
liquidity-monitor (main package)
  ├── openbb >= 4.0.0          ← fine, this installs the SDK
  └── openbb-core (implicit)   ← openbb already depends on this

liquidity.openbb_ext (provider extension layer within same package)
  └── uses openbb-core types   ← lazy-imported to avoid startup circular issues
```

**Part 2: Lazy imports inside Fetcher methods**

Per the official OpenBB documentation: "Always lazy-load imports inside functions to avoid circular imports and to speed up initial load time."

```python
# WRONG — top-level import causes circular import at module load time
from liquidity.calculators.net_liquidity import NetLiquidityCalculator

# CORRECT — lazy import inside the method
async def aextract_data(...):
    from liquidity.calculators.net_liquidity import NetLiquidityCalculator
    calc = NetLiquidityCalculator(...)
```

This is the single most important rule for avoiding circular import errors.

**Part 3: No `openbb-core` version pinning required**

Since the main package already installs `openbb>=4.0.0` (which bundles a compatible `openbb-core`), do not add a separate `openbb-core` dependency to `pyproject.toml`. The entry point registration is enough.

## 10. Testing Strategy

### Pattern 1: Fetcher `.test()` method (smoke test)

Every Fetcher has a built-in `.test()` method that runs the full TET pipeline and validates output types:

```python
# tests/openbb_ext/test_smoke.py

import pytest
from liquidity.openbb_ext.models.net_liquidity import NetLiquidityFetcher

def test_net_liquidity_fetcher_smoke():
    """Verify Fetcher runs end-to-end and returns valid types."""
    fetcher = NetLiquidityFetcher()
    # Returns None on success, raises on failure
    result = fetcher.test(
        params={"use_daily_series": False},
        credentials={"fred_api_key": None},  # None triggers fallback
    )
    assert result is None
```

### Pattern 2: vcrpy cassettes for unit tests (no live HTTP)

The official OpenBB testing approach uses `vcrpy` + `pytest-recorder` to record HTTP cassettes then replay them:

```python
# tests/openbb_ext/test_net_liquidity_fetcher.py

import pytest
from datetime import date
from openbb_core.app.service.user_service import UserService
from liquidity.openbb_ext.models.net_liquidity import NetLiquidityFetcher

# Gets credentials from OpenBB user settings file if configured
test_credentials = UserService().default_user_settings.credentials.model_dump(mode="json")

@pytest.fixture(scope="module")
def vcr_config():
    """Filter sensitive headers from cassette recordings."""
    return {
        "filter_headers": [
            ("User-Agent", None),
            ("Authorization", "MOCK_AUTH"),
        ],
        "filter_query_parameters": [("api_key", "MOCK_API_KEY")],
    }

@pytest.mark.record_http
def test_net_liquidity_fetcher(credentials=test_credentials):
    """Test NetLiquidity fetcher with recorded HTTP cassette."""
    params = {
        "start_date": date(2024, 1, 1),
        "end_date": date(2024, 3, 31),
        "use_daily_series": False,
    }
    fetcher = NetLiquidityFetcher()
    result = fetcher.test(params, credentials)
    assert result is None

# Record cassette: pytest tests/openbb_ext/test_net_liquidity_fetcher.py --record http
# Replay (CI):     pytest tests/openbb_ext/test_net_liquidity_fetcher.py
```

Cassette files live in `tests/openbb_ext/cassettes/` and SHOULD be committed to git. They make CI fast and hermetic.

### Pattern 3: Unit test the adapter logic separately

The business logic is already tested in `tests/unit/`. The adapter layer (`transform_query`, `transform_data`) is pure Pydantic validation and can be tested without HTTP:

```python
# tests/openbb_ext/test_adapters.py

from datetime import date
from liquidity.openbb_ext.models.net_liquidity import (
    NetLiquidityQueryParams,
    NetLiquidityData,
    NetLiquidityFetcher,
)

def test_transform_query_defaults():
    """Query params should apply correct defaults."""
    params = NetLiquidityFetcher.transform_query({})
    assert params.use_daily_series is False
    assert params.start_date is None

def test_transform_data_validates():
    """Data transform should validate types."""
    raw = [{
        "date": date(2024, 1, 15),
        "net_liquidity": 5200.5,
        "walcl": 7800.0,
        "tga": 1800.0,
        "rrp": 800.0,
    }]
    result = NetLiquidityFetcher.transform_data(
        NetLiquidityFetcher.transform_query({}),
        raw,
    )
    assert len(result) == 1
    assert result[0].net_liquidity == 5200.5

def test_transform_data_rejects_invalid():
    """Data transform should reject invalid types."""
    import pytest
    from pydantic import ValidationError
    raw = [{"date": "not-a-date", "net_liquidity": "not-a-float"}]
    with pytest.raises(ValidationError):
        NetLiquidityFetcher.transform_data(
            NetLiquidityFetcher.transform_query({}),
            raw,
        )
```

## 11. Phase Implementation Order (within v5.0)

Based on the architecture, the recommended phase order is:

```
Phase A: Foundation
├── Add openbb_ext/ scaffold to src/liquidity/
├── Register pyproject.toml entry point
├── Run openbb-build — verify extension loads
└── Tests: smoke test that Provider registers without import error

Phase B: Provider Extension (Python SDK)
├── NetLiquidityFetcher (wraps NetLiquidityCalculator)
├── GlobalLiquidityFetcher (wraps GlobalLiquidityCalculator)
├── StealthQEFetcher (wraps StealthQECalc)
├── Run openbb-build, test: obb.liquidity.net()
└── Tests: cassette tests for each Fetcher

Phase C: Workspace Backend
├── workspace_app.py (re-export existing FastAPI app)
├── Launch with openbb-api, connect to pro.openbb.co
├── Verify auto-generated widgets.json is correct
└── Tests: TestClient integration tests

Phase D: Router Extension (optional, Phase C+ quality)
├── Define openbb-liquidity router (if OpenBB PR desired)
├── Endpoint: obb.liquidity.net(provider="liquidity")
└── PR to OpenBB repo or publish to PyPI as openbb-liquidity
```

Phases A-C are the v5.0 scope. Phase D is v6.0 if the project goes public.

## 13. Key Architecture Risks

### Risk 1: `openbb-build` requirement after every pyproject.toml change
**Impact:** If a developer forgets `openbb-build`, the extension silently fails to register.
**Mitigation:** Add `openbb-build` to the `Makefile` and post-install hooks. Document in CLAUDE.md.

### Risk 2: Async mismatch in Fetcher
**Impact:** `NetLiquidityCalculator.calculate()` is async; Fetcher `aextract_data` must also be async. The sync `extract_data` alternative exists but blocks the event loop.
**Mitigation:** Always use `aextract_data` (not `extract_data`). The pattern in section 5 uses `aextract_data` correctly.

### Risk 3: OpenBB Data model date field type
**Impact:** OpenBB `Data` convention uses `date` (not `datetime`). Calculators return `datetime` timestamps. If you pass a `datetime` where `date` is expected, Pydantic will coerce it — but timezone info is lost.
**Mitigation:** Explicitly call `.date()` in `aextract_data` when building records dict (shown in section 5 example).

### Risk 4: vcrpy cassettes stale after API changes
**Impact:** Recorded cassettes may become stale if FRED changes response format.
**Mitigation:** Re-record cassettes quarterly or on data format changes: `pytest --record http`.

### Risk 5: Workspace backend only supports GET
**Impact:** Any POST endpoints added in the future won't be visible in OpenBB Workspace widgets.
**Mitigation:** Keep Workspace-facing endpoints as GET. POST endpoints (if added) remain internal-only.
