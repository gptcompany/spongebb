# Plan 23-03: Table Endpoint Annotations + Docker Workspace Service

## Goal

Add `openapi_extra` widget_config annotations to all existing table-compatible endpoints so OpenBB Workspace auto-discovers them as widgets, and add a Docker Compose service to run the workspace backend via `openbb-api --app` on port 6900.

## Requirements Covered

- **WS-06**: `openapi_extra` annotations on existing table endpoints with `widget_config` metadata (name, category, grid dimensions)
- **WS-07**: Docker Compose `workspace` service running `openbb-api --app` on port 6900

## Context

Plan 23-01 (wave 1) establishes version pinning, CORS origins, and the `workspace_app.py` entry point. Plan 23-02 (wave 1) adds dedicated `/workspace/*` metric and chart endpoints. This plan (wave 2) annotates the remaining existing endpoints for table widget discovery and adds the Docker infrastructure to launch the workspace.

OpenBB Workspace expects flat `list[dict]` for table widgets. Endpoints returning nested Pydantic models (e.g., `FXPairsResponse.pairs`, `CorrelationResponse.correlations`, `CalendarEventsResponse.events`) need `data.dataKey` to point Workspace at the array field. Endpoints returning flat top-level models (e.g., `StressIndicatorsResponse`, `NetLiquidityResponse`) work directly as single-row tables.

Reference: `.planning/research/features-api.md` sections 1.2-1.3 for `widget_config` schema.

### Task 8: Add Docker Compose workspace service
**File:** `docker-compose.yml`
**Action:** modify

Add `liquidity-workspace` service in the Application Services section, after `liquidity-dashboard-dev` and before `liquidity-dashboard-test`. Uses `workspace` profile so it does not start with default `docker compose up`.

```yaml
  # OpenBB Workspace backend (openbb-api --app, port 6900)
  liquidity-workspace:
    build:
      context: .
      dockerfile: Dockerfile
      target: runtime
    container_name: liquidity-workspace
    profiles:
      - workspace
    restart: unless-stopped
    ports:
      - "${LIQUIDITY_WORKSPACE_PORT:-6900}:6900"
    command: >-
      openbb-api --app liquidity.openbb_ext.workspace_app:app
      --host 0.0.0.0 --port 6900
    environment:
      - LIQUIDITY_QUESTDB_HOST=${LIQUIDITY_QUESTDB_HOST:-host.docker.internal}
      - LIQUIDITY_QUESTDB_PORT=${LIQUIDITY_QUESTDB_PORT:-9009}
      - LIQUIDITY_QUESTDB_HTTP_PORT=${LIQUIDITY_QUESTDB_HTTP_PORT:-9000}
      - LIQUIDITY_REDIS_URL=${LIQUIDITY_REDIS_URL:-redis://host.docker.internal:6379}
      - LIQUIDITY_FRED_API_KEY=${LIQUIDITY_FRED_API_KEY:-}
      - LIQUIDITY_EIA_API_KEY=${LIQUIDITY_EIA_API_KEY:-}
      - LIQUIDITY_CF_ACCESS_ENABLED=${LIQUIDITY_CF_ACCESS_ENABLED:-false}
      - LIQUIDITY_LOG_LEVEL=${LIQUIDITY_LOG_LEVEL:-INFO}
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:6900/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 20s
    networks:
      - liquidity-network
```

Key decisions:
- `target: runtime` reuses existing multi-stage build (no new Dockerfile)
- `host.docker.internal` default for QuestDB (assumes host networking to existing QuestDB)
- `profiles: [workspace]` keeps it opt-in, consistent with other profile patterns
- `start_period: 20s` accounts for OpenBB startup overhead
